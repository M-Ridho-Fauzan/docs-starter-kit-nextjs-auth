# Decision: Phase 1 — Scaffold Next.js Project

**Date**: 2026-06-25
**Mode**: Build
**Decision by**: builder + code-reviewer

## Summary

Phase 1 scaffold for the NextJS Auth Starter Kit documentation site. Migrated from plain markdown/Docusaurus-plan to Next.js 16 + Fumadocs MDX + shadcn/ui (preset `b3HawKzkye`).

## Decisions Made

### 1. Scaffold Approach
- **Decision**: Init Next.js project via `npm create-next-app` first, then add shadcn/ui manually
- **Rationale**: `npx shadcn@latest init --template next` uses create-next-app under the hood but had interactive prompts that couldn't be automated. Using create-next-app directly with `--yes` gave full control.
- **Alternatives considered**: 
  - `npx shadcn@latest init` in empty dir then copy files → chosen approach
  - `npx shadcn@latest init` directly in working dir → risked overwriting planning files
  - Manual file creation → too error-prone

### 2. shadcn/ui init failure handling
- **Decision**: Manually install shadcn dependencies + create `lib/utils.ts` + write `globals.css`
- **Rationale**: `npx shadcn@latest init --preset b3HawKzkye` timed out during "Installing dependencies" step, likely due to network/environment constraints. The `components.json` was generated correctly, so we extracted the preset config and applied it manually.
- **Dependencies installed**: class-variance-authority, clsx, tailwind-merge, lucide-react, @radix-ui/react-slot, tailwindcss-animate
- **Note**: Use `pnpm dlx shadcn@latest add <component>` for adding individual shadcn/ui components in later phases

### 3. Tailwind CSS v4 + shadcn/ui integration
- **Decision**: Use `@theme inline` block for color mappings (Tailwind v4 pattern)
- **Rationale**: Tailwind CSS v4 uses `@theme` directive instead of `tailwind.config.ts`. The `@theme inline` approach maps CSS variables to utility classes.
- **Color space**: oklch (CSS Color Level 4) — same as the starter kit project
- **CSS variables**: Defined in `:root` (light) and `.dark` (dark) following DESIGN.md specifications

### 4. globals.css structure
- **Decision**: Shadcn-style globals.css with:
  - `@import "tailwindcss"` — Tailwind v4 entry point
  - `@plugin "tailwindcss-animate"` — shadcn animation support
  - `@custom-variant dark` — dark mode variant
  - `:root` + `.dark` — CSS custom properties for both modes
  - `@theme inline` — Tailwind v4 color/font mappings
  - `@layer base` — resets and base styles
- **Rationale**: Matches shadcn/ui Tailwind v4 pattern exactly

### 5. Turbopack workspace root
- **Decision**: Set `turbopack.root: process.cwd()` in next.config.ts
- **Rationale**: The docs project shares a parent directory with the starter kit project. Without explicit root, Next.js detected the wrong lockfile and failed with a Turbopack error.
- **Impact**: `pnpm build` now passes cleanly

### 6. Font strategy
- **Decision**: Use Geist + Geist Mono from next/font/google (create-next-app defaults)
- **Rationale**: The DESIGN.md specifies Inter + JetBrains Mono, but the layout.tsx generated by create-next-app uses Geist. To avoid adding Google Font imports manually, we mapped the Geist CSS variables to Tailwind's `--font-sans` and `--font-mono`. The font can be changed in Phase 5 (Theme) without breaking anything.
- **CSS variables**: `--font-geist-sans` → `--font-sans`, `--font-geist-mono` → `--font-mono`

### 7. Review process automation
- **Decision**: After each phase, run automated code-reviewer subagent
- **Rationale**: Ensures no low/medium/critical issues remain before proceeding
- **Output**: Review documented in `.agents/reviews/phase-N-review.md`
- **Status**: Phase 1 passed with 9 medium + 2 low issues, all fixed
